# Fleet Context Engine

> Research document — how to make every worker start warm.

## Problem Statement

Every worker session starts semi-cold. The bridge manually composes context, but:

1. **No codebase map.** The bridge doesn't know which files relate to which features. It includes what it remembers or can grep for, but misses implicit connections (a model changed → serializer, controller, tests, and API docs all affected).
2. **Session digests are flat.** `.fleet/sessions/` accumulates markdown, but there's no way to query "what sessions touched authentication?" without reading every file.
3. **Knowledge is unindexed.** `.fleet/knowledge/` grows over time but entries aren't tagged, weighted, or connected to specific files or features.
4. **Cross-worker context is message-passing.** `_bridge/messages/` works for directed communication, but worker A discovering a gotcha in `OrderService` doesn't automatically inform worker B who's about to touch `OrderService`.
5. **No relevance ranking.** When context is included in a prompt, everything gets equal weight. A 200-line architecture doc is included wholesale even when the worker only needs 3 lines about the payment module.

## Design Principles

- **Plain files, no database.** Fleet's identity is simple files. The context engine uses JSON and markdown — nothing that requires a running process.
- **Bridge-side only.** Workers don't run the context engine. The bridge assembles context before spawning. Workers contribute to it passively through session digests and discoveries.
- **Lazy computation.** Indexes are built on demand and cached. No background daemon.
- **Token-aware.** Every output is designed for inclusion in an LLM prompt. Summaries are default; verbatim inclusion is opt-in.

## 1. Codebase Index

### What it captures

A lightweight map of the codebase stored at `.fleet/index/codebase.json`. Not a full AST — a file-level graph with enough signal for relevance matching.

### Data structure

```json
{
  "version": 1,
  "generated_at": "2026-04-10T12:00:00Z",
  "commit": "abc123",
  "files": {
    "app/models/order.rb": {
      "type": "model",
      "feature": ["orders", "checkout"],
      "imports": ["app/models/line_item.rb", "app/models/payment.rb"],
      "imported_by": ["app/controllers/orders_controller.rb", "app/services/checkout_service.rb"],
      "tests": ["spec/models/order_spec.rb"],
      "last_modified": "2026-04-08",
      "change_frequency": "high",
      "summary": "Order aggregate root. Has many line_items and one payment. Handles state machine for order lifecycle."
    }
  },
  "features": {
    "checkout": {
      "files": ["app/models/order.rb", "app/services/checkout_service.rb", "app/controllers/checkout_controller.rb"],
      "entry_points": ["app/controllers/checkout_controller.rb"],
      "description": "End-to-end checkout flow from cart to confirmation"
    }
  },
  "dependency_clusters": [
    {
      "name": "payment-processing",
      "files": ["app/services/payment_service.rb", "app/models/payment.rb", "lib/stripe_client.rb"],
      "external_deps": ["stripe"]
    }
  ]
}
```

### How it's built

The bridge generates the index by asking a dedicated worker (or doing it inline for small repos):

1. **File inventory.** `find` + `git ls-files` to get all tracked files with modification dates.
2. **Import graph.** Language-aware regex extraction of imports/requires. Not perfect — but good enough for relevance. For ambiguous languages (Ruby, Python), fall back to `grep` for class/module references.
3. **Feature detection.** Heuristic clustering based on:
   - Directory structure (most codebases group by feature or layer)
   - Naming conventions (`checkout_*.rb`, `*_payment_*`)
   - Co-change frequency from `git log --name-only` (files that change together likely belong together)
4. **Summaries.** One-line description per file, generated by reading the first few lines + class/function signatures. For large repos, only summarize files with high change frequency.

### When it's updated

- **On first use** in a project: full generation.
- **Before each dispatch batch**: incremental update. `git diff --name-only ${last_commit}..HEAD` identifies changed files. Only those files and their immediate dependents get re-indexed.
- **Staleness check**: If `codebase.json` exists but `commit` doesn't match current HEAD, the bridge runs an incremental update.

### Cost

For a 500-file repo, the initial index takes one worker ~2 minutes. Incremental updates take seconds (just re-index changed files). The JSON file itself is typically 50-200KB — negligible.

## 2. Automatic Context Assembly

### The assembly pipeline

Given a task description, the engine produces a **context package** — a structured markdown document included in the worker's prompt.

```
Task description
       │
       ▼
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  File-level  │────▶│ Feature-level│────▶│ History-level│
│  relevance   │     │  expansion   │     │  enrichment  │
└─────────────┘     └──────────────┘     └──────────────┘
       │                    │                    │
       ▼                    ▼                    ▼
┌─────────────────────────────────────────────────────┐
│              Token budget allocation                 │
└─────────────────────────────────────────────────────┘
       │
       ▼
  Context package (markdown)
```

### Step 1: File-level relevance

Extract signals from the task description:
- **Explicit file mentions**: "fix the bug in `OrderService`" → `app/services/order_service.rb`
- **Class/method mentions**: "the `process_payment` method" → grep the index for which file contains it
- **Error traces**: stack trace lines → file paths
- **Issue/PR references**: `gh issue view 42 --json body` → extract file references from the issue body

Score each file 0-1 based on signal strength. Explicit mention = 1.0, import neighbor = 0.6, same feature = 0.4, recent co-change = 0.3.

### Step 2: Feature-level expansion

Using the `features` map from the index, if any matched file belongs to a feature, pull in other files from that feature at reduced relevance (0.3-0.4). This catches the common case where a bug in `order.rb` also needs the worker to know about `checkout_service.rb`.

Don't expand features that are too large (>20 files). Instead, include the feature description and let the worker explore.

### Step 3: History-level enrichment

Query recent context:
- **Session digests** that reference matched files (see §3)
- **Discoveries** related to matched features
- **Knowledge entries** (architecture, conventions, gotchas) that mention matched files or features
- **Recent git activity**: `git log --oneline -10 -- {matched_files}` to show what changed recently and why

### Step 4: Token budget allocation

The engine works within a configurable token budget (default: 8,000 tokens for context, leaving plenty of room for the task prompt and worker protocol).

Priority tiers for budget allocation:

| Tier | Content | Format | Budget share |
|------|---------|--------|-------------|
| 1 | Task-specific files + error traces | Verbatim snippets | 40% |
| 2 | Relevant conventions + gotchas | Verbatim | 20% |
| 3 | Feature map + dependency context | Summarized | 15% |
| 4 | Session history + discoveries | Summarized | 15% |
| 5 | Architecture overview | Summarized | 10% |

If tier 1 content alone exceeds its budget, lower tiers get compressed further. If tier 1 is small (simple task), lower tiers get more room.

### Output format

```markdown
## Context Package for: {worker-name}

### Relevant Files
- `app/services/order_service.rb` — Order processing logic. **Direct match.**
- `app/models/order.rb` — Order model, state machine. Related via checkout feature.
- `spec/services/order_service_spec.rb` — Existing test coverage.

### Key Context
[Verbatim snippets from the most relevant files — the specific method, the error site, etc.]

### Conventions
- Services follow the command pattern: `call` class method, return Result object.
- All monetary values stored as cents (integer).

### Gotchas
- OrderService uses optimistic locking. Tests must reload records after save.

### Recent History
- Session `fix-payment-race` (2026-04-08): Fixed race condition in concurrent payment processing. Changed locking strategy in OrderService.
- `abc123` (2026-04-09): "Add retry logic to Stripe webhook handler"

### Feature Map: checkout
Entry point: `CheckoutController#create` → `CheckoutService#call` → `OrderService#process` → `PaymentService#charge`
```

## 3. Session Memory

### Structured digests

Replace the current freeform `.fleet/sessions/{name}.md` with a structured format that supports querying:

```json
{
  "session": "fix-payment-race",
  "completed_at": "2026-04-08T16:30:00Z",
  "outcome": "PR #84 created, merged",
  "tags": ["payments", "concurrency", "orders"],
  "files_touched": [
    "app/services/order_service.rb",
    "app/services/payment_service.rb",
    "spec/services/order_service_spec.rb"
  ],
  "features": ["checkout", "payments"],
  "decisions": [
    {
      "what": "Used pessimistic locking instead of optimistic",
      "why": "Optimistic locking caused silent failures under concurrent Stripe webhooks"
    }
  ],
  "discoveries": [
    "Stripe webhooks can arrive before the initial API response returns, causing a race on order state transitions"
  ],
  "summary": "Fixed race condition where concurrent Stripe webhook delivery could double-process a payment. Switched OrderService from optimistic to pessimistic locking for the payment state transition."
}
```

The human-readable markdown digest (`.fleet/sessions/{name}.md`) continues to exist — it's generated from this structured data. Both are committed. The JSON is for machine consumption; the markdown is for humans browsing the repo.

### Index file

`.fleet/index/sessions.json` maintains a lightweight index:

```json
{
  "sessions": [
    {
      "name": "fix-payment-race",
      "completed_at": "2026-04-08T16:30:00Z",
      "tags": ["payments", "concurrency", "orders"],
      "files_touched": ["app/services/order_service.rb", "app/services/payment_service.rb"],
      "features": ["checkout", "payments"],
      "summary": "Fixed payment race condition via pessimistic locking"
    }
  ]
}
```

### Querying

The bridge queries session memory during context assembly:

```
relevant_sessions = sessions.filter(s =>
  s.files_touched.intersects(matched_files) ||
  s.features.intersects(matched_features) ||
  s.tags.intersects(task_tags)
).sort_by(s => s.completed_at).reverse().take(5)
```

Most recent sessions that touched the same files or features get included in the context package. Older sessions get progressively summarized (full detail for last 3, one-liner for older).

## 4. Cross-Worker Context

### The problem with message-passing

The current `_bridge/messages/{session}.md` is point-to-point. Worker A writes a discovery, the bridge notices, and manually writes it to worker B's message file. This requires the bridge to be actively monitoring and to understand which discoveries affect which workers.

### Solution: Discovery index with automatic routing

#### Discovery structure

When a worker writes to `_bridge/discoveries/{topic}.md`, it also writes a structured sidecar:

```json
// _bridge/discoveries/{topic}.json
{
  "topic": "stripe-webhook-ordering",
  "discovered_by": "fix-payment-race",
  "discovered_at": "2026-04-08T15:00:00Z",
  "affects_files": ["app/services/payment_service.rb", "app/controllers/webhooks_controller.rb"],
  "affects_features": ["payments", "webhooks"],
  "severity": "high",
  "summary": "Stripe webhooks can arrive before the initial API response. Any code touching payment state must handle this race."
}
```

#### Automatic routing

The bridge maintains a **worker-file mapping** in `_bridge/graph.json` (extending the existing dependency graph):

```json
{
  "nodes": [
    {
      "id": "fix-payment-race",
      "status": "active",
      "files_scope": ["app/services/order_service.rb", "app/services/payment_service.rb"],
      "features_scope": ["checkout", "payments"]
    },
    {
      "id": "add-webhook-retry",
      "status": "active",
      "files_scope": ["app/controllers/webhooks_controller.rb", "app/jobs/webhook_retry_job.rb"],
      "features_scope": ["webhooks"]
    }
  ]
}
```

When a new discovery is written, the bridge checks if any active worker's `files_scope` or `features_scope` overlaps with the discovery's `affects_files` or `affects_features`. If so, it automatically writes the discovery summary to that worker's message file.

This is reactive, not proactive — it happens when the bridge processes a status update or discovery, not via a background daemon. The bridge's monitoring loop (the `while true; sleep 15` background task) is the natural trigger.

#### Conflict detection

The worker-file mapping also enables early conflict detection. Before dispatching, the bridge checks:

```
for each new_worker:
  overlap = new_worker.files_scope ∩ active_workers.files_scope
  if overlap is non-empty:
    warn bridge ("Worker {new} and {existing} both touch {files}")
```

This moves conflict detection from "after the fact" to "before dispatch."

## 5. Progressive Enrichment

### What signals improve the engine

| Signal | Source | What it improves |
|--------|--------|-----------------|
| Session outcomes | Session digests | Feature boundaries, file groupings |
| Worker file access patterns | Session `files_touched` | Import graph accuracy, feature detection |
| Discoveries that become knowledge | Bridge promotion flow | Gotchas and conventions quality |
| Context that workers ignored | Diff between context given and files actually touched | Relevance scoring |
| Context workers had to find themselves | Files touched that weren't in the context package | Coverage gaps |
| Director corrections | "No, that's not related to checkout" | Feature boundaries |

### Feedback loop

After each session completes:

1. **Compare context given vs. context used.** If the worker was given 10 files in the context package but only touched 3, the other 7 had inflated relevance scores. Decrease their feature association strength.
2. **Identify context gaps.** If the worker touched 5 files not in the context package, those files should have been included. Strengthen their association with the task's features/files.
3. **Update feature boundaries.** If a worker working on "checkout" also touched `app/mailers/order_mailer.rb`, that file should be associated with the checkout feature.
4. **Promote discoveries.** If the same gotcha appears in 2+ session discoveries, the bridge should promote it to `.fleet/knowledge/gotchas.md`.

### Bounded growth

Without bounds, the index grows forever. Controls:

- **Session retention window.** Keep full detail for the last 20 sessions. Older sessions keep only their index entry (tags, files, one-line summary). The full JSON/markdown is archived to `.fleet/sessions/archive/`.
- **Relevance decay.** File association scores decay over time. A co-change from 6 months ago is less relevant than one from last week. Use a simple exponential decay: `score * 0.95^(weeks_since_event)`.
- **Feature pruning.** If a feature hasn't been referenced in any session for 30+ days, mark it `stale` in the index. Stale features are excluded from context assembly unless explicitly mentioned in the task.
- **Index size cap.** If `codebase.json` exceeds 500KB, drop summaries for low-change-frequency files and reduce `imported_by` lists to top-5.

## 6. Token Efficiency

### The budget system

Every context package has a token budget. The bridge can configure this, but defaults are:

| Component | Default budget | Notes |
|-----------|---------------|-------|
| Worker protocol | ~2,000 tokens | Fixed, always included |
| Task prompt | ~2,000 tokens | The actual task description |
| Context package | ~8,000 tokens | Assembled by the engine |
| **Total overhead** | **~12,000 tokens** | Leaves 180K+ for the worker's own exploration |

### Compression strategies

**Tier 1 — Verbatim (highest value):**
- The specific code snippet where the bug lives
- Error output / stack traces
- Relevant test files (just the test cases, not setup boilerplate)

**Tier 2 — Structured summary:**
- Feature maps: file list + one-line descriptions + entry point chain
- Session history: one paragraph per relevant session
- Convention/gotcha entries: verbatim (they're already concise)

**Tier 3 — Compressed reference:**
- Architecture overview: just the section relevant to the task
- Dependency clusters: names only, not full file lists
- Older session history: "3 prior sessions touched this area (see .fleet/sessions/)"

**Tier 4 — Pointer only:**
- "Full architecture docs at .fleet/knowledge/architecture.md"
- "12 prior sessions in this area — most recent: fix-payment-race (2026-04-08)"

### Adaptive budgeting

Not all tasks need the same context distribution:

| Task type | Heavy on | Light on |
|-----------|----------|----------|
| Bug fix | Code snippets, error output, test files | Architecture, feature maps |
| Feature | Architecture, conventions, feature maps | Specific code (worker will explore) |
| Refactor | Dependency map, test coverage, conventions | Session history |
| Research | Architecture, prior sessions, external links | Code snippets |

The engine adjusts budget allocation based on the work type detected from the task description.

## Concrete Examples

### Example A: Bug fix task

**Task:** "OrderService#process raises NoMethodError on nil payment when a guest checks out without saving payment details first."

**Context assembly trace:**

1. **File-level relevance:**
   - `app/services/order_service.rb` — explicit mention (score: 1.0)
   - `app/models/order.rb` — imported by OrderService (score: 0.6)
   - `app/models/payment.rb` — mentioned in task ("payment") (score: 0.7)
   - `spec/services/order_service_spec.rb` — test file for match (score: 0.8)

2. **Feature expansion:**
   - "checkout" feature includes all above + `app/controllers/checkout_controller.rb` (score: 0.4)
   - "guest-checkout" sub-feature if detected (score: 0.5)

3. **History enrichment:**
   - Session `fix-payment-race` touched `order_service.rb` — include its summary
   - Discovery `stripe-webhook-ordering` affects payments — include
   - `git log -5 -- app/services/order_service.rb` shows recent changes

4. **Token budget (8,000 tokens):**
   - Tier 1 (3,200 tokens): `OrderService#process` method verbatim (~40 lines), the Payment model's relevant associations, the existing test file for `#process`
   - Tier 2 (1,600 tokens): Convention about nil-guarding in services, gotcha about optimistic locking
   - Tier 3 (1,200 tokens): Checkout feature map (entry point chain), session `fix-payment-race` summary
   - Tier 4 (800 tokens): Architecture pointer, 2 older session one-liners

**Resulting context package:**

```markdown
## Context Package for: fix-guest-checkout-npe

### Direct Context
`app/services/order_service.rb:45-82` — the `#process` method:
[verbatim code block]

`app/models/payment.rb:1-15` — Payment model associations:
[verbatim code block]

### Tests
`spec/services/order_service_spec.rb` — existing test cases for `#process`:
[verbatim test block — 3 test cases]

### Conventions
- Services return `Result.new(success: bool, error: msg)`. Don't raise from service methods.
- Nil-guard external dependencies. Guest checkout means `order.payment` can be nil before the payment step.

### Gotchas
- OrderService uses pessimistic locking since fix-payment-race. Tests must use `reload` after saves.
- Stripe webhooks can arrive before API response — don't assume payment exists just because checkout started.

### Recent Changes to OrderService
- `abc123` (2026-04-08): "Switch to pessimistic locking for payment transitions"
- `def456` (2026-04-06): "Add guest checkout flag to Order model"

### Feature: checkout
Entry point: `CheckoutController#create` → `CheckoutService#call` → `OrderService#process`
Guest flow skips `PaymentDetailsController` — payment is created inline during `#process`.
```

    {topic}.md             # Existing — human-readable

## Implementation Sequence

This can be built incrementally. Each step adds value independently:

1. **Structured session digests.** Change the worker protocol to output JSON alongside markdown. Update the bridge skill to create/maintain `sessions.json` index. Immediate value: bridge can query "what sessions touched X?"

2. **Codebase index generator.** Build a script or worker prompt that generates `codebase.json`. Run it once per project. Immediate value: bridge has a file-level map for prompt composition.

3. **Context assembly in the bridge skill.** Add a "compose context" step to the bridge's dispatch flow. Read the index, match against the task, produce a context package. Immediate value: workers start with targeted, relevant context instead of everything-or-nothing.

4. **Discovery routing.** Add JSON sidecars to discoveries. Add overlap detection to the bridge's monitoring loop. Immediate value: cross-worker context flows automatically.

5. **Feedback loop.** After sessions complete, compare context-given vs. files-touched. Update relevance scores. Immediate value: context assembly gets more accurate over time.

6. **Token budgeting.** Add explicit budget tiers to the assembly pipeline. Immediate value: consistently right-sized context regardless of codebase size.

## Key Design Decisions

**Why JSON, not a vector database?** Fleet's identity is plain files with no running services. A vector DB would give better semantic search, but it adds operational complexity (process management, embedding model dependency). The structured JSON approach with keyword/file-path matching handles 90% of use cases. If semantic search becomes necessary later, it can be added as an optional enhancement — generate embeddings and store them in JSON alongside the index.

**Why bridge-side assembly, not worker self-service?** Workers that assemble their own context burn tokens on exploration instead of their actual task. The bridge has global visibility (all workers, all sessions, all discoveries) and can compose context once, cheaply. Workers still have full exploration capability for edge cases the engine didn't anticipate.

**Why file-level, not function-level indexing?** Function-level indexing requires language-specific AST parsing for every supported language. File-level indexing works with regex and git, which Fleet already uses. The import graph captures most cross-file dependencies. Function-level detail is included verbatim in the context package where it matters (the specific method for a bug fix) — the index just needs to know which file to pull from.

**Why exponential decay, not manual curation?** Manual curation doesn't scale. Exponential decay means the engine naturally prioritizes recent, relevant context without anyone having to prune. The 20-session retention window and feature staleness checks provide hard bounds.
